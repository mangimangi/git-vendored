# git-vendored — image-baked medici (phase 4.5)

> Self-contained per-repo planning doc. You do **not** need to read the parent medici epic. All context you need is below.

## Why this work exists

A sibling orchestrator project (medici) has migrated its toolchain from per-repo vendored copies into a baked container image. As of phase 4 of that migration:

- Target repos no longer carry `.vendored/` at all — no `pkg/`, no `configs/`, no `manifests/`, no `hooks/`
- Session hooks are baked into the medici runtime image at a known path; nothing per-repo wires them up at session start
- Tools (`prl`, `madp`) are `pip install`'d into the image at build time and live on `PATH`

That migration leaves a chunk of git-vendored code with **zero remaining consumers** — the entire session-hook orchestration subsystem. This work removes that dead subsystem cleanly.

**This is not a sunset.** git-vendored's core function — fetching git-based tool releases and installing them at known paths on a host — is still actively used to install `git-semver` and `git-dogfood` on developer workstations. Those are git plugins, not Python packages, and `pip` cannot install them. git-vendored remains the right tool for that job.

## Scope

### Delete (no remaining consumers)

- **`templates/.vendored/hooks/`** — session hook templates (e.g., `vendored-session.sh`). No target repo will use them after phase 4.
- **Hook installation logic** — any code in the install command that wires hooks into a target repo's `.claude/settings.json` or writes files into `.vendored/hooks/`. Both target paths are gone.
- **Session-time install path** — any code path designed to run vendoring *inside a session container at startup*. Sessions no longer install vendored tools at start; everything is baked into the image.
- **Per-repo session-time health checks** — anything that ran from `.vendored/hooks/` to verify install state at SessionStart. The medici image now has its own `which prl && which madp` health check in `session.sh`. Anything in git-vendored that exists only to support that runtime hook flow goes away.
- **Tooling-fixes P0 (dynamic shim resolution)** — moot. Never formally tracked as a prl issue; noted here as resolved by the image-baked migration (no shims at session time).
- **Tooling-fixes P5 (hook failure handling)** — moot for the same reason. Never formally tracked; resolved by removing session hooks entirely.
- **Epic gv-3adc (Vendored Session Hooks)** — entirely obsolete now that session-hook orchestration is removed. Its children (gv-3adc.1, gv-3adc.2) were implemented but the feature they built has been deleted.
- **Anything in `.github/workflows/` that wires hooks or runs hook-related checks** — verify and remove. Workflows that run `.vendored/check` or `.vendored/audit` against the install state of the *core vendoring* still apply; only the hook-specific bits go.

### Keep (still used for git-semver / git-dogfood on host)

- **Core vendoring command** — `git-vendored install <tool>@<version>`, the fetch-from-repo + install-files-at-paths machinery. This is exactly what installs `git-semver` and `git-dogfood` onto developer workstations and CI runners that build the medici fork.
- **Manifests** — `.vendored/manifests/<tool>.version`, `.post-installed`, etc. Track host-side install state. Still needed.
- **Audit / `.vendored/check`** — verifies installed state matches expected. Still needed for host-side installs.
- **`.vendored/config.json`** schema and validation — still needed for the surviving install flows.
- **`templates/`** — keep the *non-hook* templates if any exist (e.g., `.vendored/install`, `.vendored/check`, `.vendored/audit` shell shims). Only the `hooks/` subtree under `templates/` is dead.
- **Documentation** — update README and AGENTS.md to reflect the narrowed surface area, but keep the docs that describe the surviving functionality.

### ~~Pre-existing bug to fix as part of this work~~ — DESCOPED

**Install workflow does not recursively copy package subdirectories.** Discovered during the medici phase 3 dogfood (2026-04-08): madreperla v0.0.9 was the first release to ship modes as package data at `madreperla/modes/*.md`. The vendored install dropped the `modes/` subdirectory silently, breaking madp's mode resolution.

**DESCOPED 2026-04-09.** The architectural direction is to **retire git-vendored entirely** in favor of pip-installed tools (the image-baked model from `mdci-cf3d`). The recursive-copy bug only affects the vendored model. Fixing it is wasted effort — pip handles package data correctly out of the box, and after phase 4 ships, no consumer will use the vendored install path for python packages anyway.

**What this means for phase 4.5:**

- **Drop** the recursive package-data copy fix from the scope of this epic (was `gv-XXXX.1`)
- **Drop** the regression test against madreperla v0.0.9's modes/ subdirectory
- **Keep** the rest of the epic as-is: delete the dead session-hook subsystem, strip hook installation logic, close obsolete tooling-fixes P0/P5, update docs, regression-verify host-side `git-vendored install git-semver` and `git-dogfood` (which are NOT python packages and don't have the package-data issue)

**Where the workaround lives in the meantime:** for any repo that needs working vendored madp during the transition (i.e., before phase 4 ships and target repos drop `.vendored/`), the workaround is the per-repo hand-cp of `madreperla/modes/` into `.vendored/pkg/madreperla/modes/`. We did this once for the medici repo (commit `5927170`). After phase 4, no repo needs it.

**The "right direction" path** (decided 2026-04-09 during mp-3b45 dogfood): use the new image-baked path for any work that needs working mode prompt injection. The new image has pip-installed madp with package data correctly bundled. The vendored path is for legacy / transitional use only.

**Why fix it here even though we're sunsetting the session-hook subsystem:** the install command itself isn't going away — it's still needed for git-semver / git-dogfood on host, and it's still the canonical way the medici fork itself bakes tools (until phase 3 of mdci-cf3d ships and switches to pip). Until that switch is complete, this bug will keep biting any release that grows package data. Worth one well-tested commit before phase 4.

### Out of scope

- **Schema framework** (the `mdci-b452` epic in medici / git-vendored). That's a separate, parallel piece of work with its own per-repo planning. Do not touch it here.
- **Sunset decision.** Do not decide whether git-vendored should eventually be replaced by something else. That's a future conversation. This work assumes git-vendored stays maintained for its narrowed scope.
- **Rewriting the install command.** No redesign — only deletion of code paths that have lost their consumers, plus the recursive-copy bug fix above.

## Acceptance criteria

1. `templates/.vendored/hooks/` no longer exists in git-vendored
2. The install command, when run against a target repo, no longer attempts to write files into `.vendored/hooks/` or modify `.claude/settings.json`
3. Searching the repo for references to `vendored-session.sh`, `SessionStart`, or `.claude/settings.json` returns no production code (only historical references in commits and changelog entries are acceptable)
4. **Host-side install of git-semver still works end-to-end:** in a clean environment, `git-vendored install git-semver@<version>` produces a working `git semver` command. Same for `git-dogfood`. This is the regression-protection check — if either of these breaks, we removed too much.
5. README and AGENTS.md reflect the narrowed scope and explicitly list session-hook orchestration as removed (not just absent — readers should know it was intentionally deleted, not forgotten)
6. The git-vendored test suite passes after deletions; any tests that were exercising the dead session-hook code paths are removed (not skipped)
7. If `mdci-b452` (schema framework epic) had any cross-references to the deleted hook code, they are updated or noted

## Sequencing — depends on phase 4

This work **must run after phase 4** of mdci-cf3d (target repo migrations to the image-baked model). Reason: phase 4 is what removes `.vendored/` from the consumer repos. Until that's done, technically there's still production code paths reading the hook subsystem in target repos, and deleting the source side first would leave half-migrated state.

Specifically, do not start this work until:
- All listed target repos in mdci-cf3d phase 4 have been migrated and smoke-tested against the new image
- The medici fork's own image build (phase 3) is shipped and stable
- `git grep` across the workspace's target repos shows no references to `.vendored/hooks/` or vendored-session.sh

Once that's true, this work is purely deletion + verification.
