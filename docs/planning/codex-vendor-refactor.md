# Codex Support & Vendor DX Improvements

Three related improvements to the git-vendored framework:

1. **Multi-agent hook orchestration** — generate agent-specific config for Claude Code and Codex
2. **Standard download helpers** — reduce boilerplate in vendor `install.sh` scripts
3. **Vendor support/feedback config** — let tools declare how to submit bug reports

---

## 1. Multi-Agent Hook Orchestration

### Problem

Session hook orchestration is currently owned by madreperla (`--install-hooks`),
which generates `.claude/settings.json` and `.claude/hooks/vendored-session.sh`.
This is misplaced — git-vendored owns the hook lifecycle (`--list`, `--hooks`)
and should own the agent config files too. madreperla should focus on prompt
content, not hook plumbing.

Additionally, the orchestrator and vendor hooks reference `$CLAUDE_PROJECT_DIR`
(a Claude Code env var), making them non-portable.

### Scope

**In scope (git-vendored):**

- New command: `python3 .vendored/install --setup-hooks [agent]`
  - `agent` is `claude` or `codex` (default: auto-detect based on which dotdir exists, or both)
  - For Claude: generates `.claude/settings.json` with SessionStart/resume hooks
  - For Codex: generates `.codex/config.toml` with SessionStart hook (format TBD, see below)
  - Both point to the same orchestrator script (`.vendored/hooks/vendored-session.sh`)
- Move `vendored-session.sh` from `.claude/hooks/` to `.vendored/hooks/`
  - Replace `$CLAUDE_PROJECT_DIR` fallback with `git rev-parse --show-toplevel`
  - Keep `$PROJECT_DIR` as the primary (set by the framework when invoking hooks)
- The existing `vendored-session.sh` template (currently generated by madreperla)
  becomes a framework-owned file, tracked in the git-vendored manifest
- Update `install.sh` (bootstrap) to run `--setup-hooks` after initial install

**Out of scope (vendor repo prereqs — noted for tracking):**

- **pearls**: 4 shell hooks (`session-start.sh`, `session-resume.sh`,
  `post-install.sh`, `pre-push`) each have
  `PROJECT_DIR="${PROJECT_DIR:-${CLAUDE_PROJECT_DIR:-...}}"`.
  Change to `PROJECT_DIR="${PROJECT_DIR:-$(git rev-parse --show-toplevel)}"`.
  One-line change per file. No functional change — `$PROJECT_DIR` is already
  set by the orchestrator in practice.

- **madreperla**: Delete hook generation code entirely:
  - Remove `--install-hooks` flag and `write_settings_json()`
  - Remove `_build_vendored_entries()` and `ORCHESTRATOR_SCRIPT` constant
  - Remove `VENDORED_MARKER` constant
  - Remove "hi claude" from `get_prompt_intro()` (parameterize or remove)
  - Keep all prompt generation logic — that's madreperla's job

### Codex Hook Config Format

**Status: TBD / experimental.** Codex added SessionStart + Stop hooks in
v0.114.0 (experimental). The format mirrors Claude Code's approach — a command
runs at session start, stdout is injected into context before the first turn.

Config is TOML-based in `.codex/config.toml` (project-level).

Relevant references:
- [Codex hooks PR #13276](https://github.com/openai/codex/pull/13276)
- [SessionStart hook issue #13014](https://github.com/openai/codex/issues/13014)
- [Codex Advanced Configuration](https://developers.openai.com/codex/config-advanced)
- [Codex Configuration Reference](https://developers.openai.com/codex/config-reference)

The exact TOML schema for hooks is experimental and may change. The
`--setup-hooks codex` implementation should be built against the stable format
once it ships, or behind a flag while experimental.

### Design Notes

- `--setup-hooks` is idempotent — safe to re-run after vendor install/remove
- Agent config files (`.claude/settings.json`, `.codex/config.toml`) are
  **not** in any vendor's manifest — they're project-owned, framework-generated
- The orchestrator script IS in git-vendored's manifest (framework-owned)
- If neither `.claude/` nor `.codex/` exists, `--setup-hooks` with no argument
  is a no-op (don't create agent dirs speculatively)

## 2. Standard Download Helpers

### Problem

Every vendor's `install.sh` implements its own curl logic for downloading files
from GitHub. Common patterns duplicated across vendors:

- GitHub API authentication (`Authorization: token $GH_TOKEN`)
- Downloading raw files by path and ref
- Error handling / HTTP status checking
- Writing to `$VENDOR_INSTALL_DIR`
- Making scripts executable

### Proposal

Ship a shell library that `install.sh` can source:

```bash
# vendor's install.sh
source "$VENDOR_LIB" 2>/dev/null || {
    # fallback: inline the helpers (backwards compat with older framework)
    fetch_file() { curl -fsSL ...; }
}

fetch_file "src/cli.py" "$VENDOR_INSTALL_DIR/cli.py"
fetch_file "src/hooks/session-start.sh" "$VENDOR_INSTALL_DIR/hooks/session-start.sh" +x
```

**`$VENDOR_LIB`** — new env var passed by the framework, pointing to
`.vendored/lib/vendor-helpers.sh`. Contains:

- `fetch_file <repo_path> <local_path> [+x]` — download a file from
  `$VENDOR_REPO` at `$VENDOR_REF`, optionally chmod +x
- `fetch_dir <repo_path> <local_path>` — download a directory tree
- Auth handled internally using `$GH_TOKEN` / `$VENDOR_PAT`
- Writes downloaded paths to `$VENDOR_MANIFEST` automatically

### Benefits

- Vendors don't implement auth, error handling, or manifest writing
- Framework can optimize (parallel downloads, caching, rate limit handling)
- Manifest is always correct (no vendor bugs in manifest emission)
- Reduces vendor `install.sh` to ~5 lines for typical tools

### Backwards Compatibility

- `$VENDOR_LIB` is a new env var — old frameworks don't set it
- Vendors should include a fallback block (shown above) for graceful degradation
- Existing `install.sh` scripts continue to work unchanged
- Migration is opt-in per vendor

### Future: Declarative File Lists

A natural evolution would be declaring files in config instead of `install.sh`:

```json
{
  "_vendor": {
    "files": [
      {"src": "src/cli.py", "dst": "cli.py"},
      {"src": "src/hooks/session-start.sh", "dst": "hooks/session-start.sh", "executable": true}
    ]
  }
}
```

If `_vendor.files` is present, the framework downloads them directly — no
`install.sh` needed. This covers the common case (download N files from GitHub).
Vendors with build steps or non-GitHub sources still use `install.sh`.

**Not proposed for this iteration** — the helpers approach is lower-risk and
backwards compatible. Declarative lists can build on the same `fetch_file`
internals later.

## 3. Vendor Support / Feedback Config

### Problem

When something goes wrong with a vendored tool, there's no standard way to know
where to report bugs or request features. Each tool has its own repo, issue
tracker, and conventions, but this information isn't surfaced.

### Proposal

#### Config Schema

Add an optional `support` key to `_vendor`:

```json
{
  "_vendor": {
    "repo": "mangimangi/pearls",
    "support": {
      "issues": "https://github.com/mangimangi/pearls/issues",
      "instructions": "Include the output of `prl list` and your .pearls/issues.jsonl in bug reports.",
      "labels": ["bug", "feature-request"]
    }
  }
}
```

**Smart default:** if `_vendor.support` is absent, derive from `_vendor.repo`:
- `issues` → `https://github.com/<repo>/issues`
- `instructions` → empty
- `labels` → empty

#### Standalone Feedback Script

A separate script at `.vendored/feedback` (not a flag on install):

```bash
python3 .vendored/feedback                # all vendors
python3 .vendored/feedback pearls         # specific vendor
```

Output (structured, parseable):

```
=== pearls (mangimangi/pearls) ===
Issues: https://github.com/mangimangi/pearls/issues
Labels: bug, feature-request
Instructions: Include the output of `prl list` and your .pearls/issues.jsonl in bug reports.

=== git-semver (mangimangi/git-semver) ===
Issues: https://github.com/mangimangi/git-semver/issues
```

This script is framework-owned (in git-vendored's manifest), installed
alongside `install`, `check`, and `remove`.

#### Context Injection (Light Touch)

By default, the framework injects a **breadcrumb** into session context — not
the full support details. This keeps context lean while making support
discoverable:

> Installed tools with feedback loops: pearls, git-semver, git-vendored.
> Run `.vendored/feedback <tool>` for details.

This breadcrumb is emitted by the orchestrator (vendored-session.sh) during
session start — it's stdout that gets injected into the agent's context
alongside vendor session-start hook output.

When a tool misbehaves, the agent already knows how to look up full details.

#### Vendor-Provided Detail

Tools that want richer support docs can provide them in their own AGENTS.md or
session-start hook output. The framework's job is the breadcrumb + query
command; the tool's job is the detailed "how to file a good bug report" content
if it wants to go beyond the config fields.

### Open Questions

- Should `feedback` output be JSON for machine consumption, or human-readable
  with a `--json` flag?
- Should the breadcrumb list ALL vendors or only those with explicit `support`
  config? (Leaning: all, since smart default covers every vendor.)

## Sequencing

```
1. Standard download helpers ($VENDOR_LIB)
   - No dependencies, smallest scope
   - Enables simpler vendor install.sh going forward

2. Vendor support/feedback config (_vendor.support + .vendored/feedback)
   - No dependencies on (1) or (3)
   - Small, self-contained

3. Multi-agent hook orchestration (--setup-hooks)
   - Prereqs in vendor repos (pearls, madreperla) should land first
   - madreperla must drop hook generation before git-vendored takes it over
   - Can start git-vendored side in parallel with vendor prereqs

All three are independent and can be developed in parallel.
```
