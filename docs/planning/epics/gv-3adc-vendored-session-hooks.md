# Vendored Session Hooks — Framework-Level Hook Lifecycle

**Status:** Planning → Ready for Refinement
**Priority:** P1
**Epic:** gv-3adc

## Problem

Vendored packages that need session-time configuration (Claude Code session start/resume) have no framework-level mechanism. Today pearls manually templates `.claude/hooks/configure.sh` and `.claude/settings.json` into consumer repos via `install.sh`. This doesn't scale — if git-semver, git-dogfood, or any other vendored tool wanted session hooks, they'd each independently write to `.claude/settings.json` and `.claude/hooks/`, causing conflicts.

Additionally, pearls' `configure.sh` mixes two concerns:
- **Post-install setup** (merge driver registration, pre-push hook symlinks) — needs `.git/` access, only needs to run once per install
- **Session setup** (PATH shim creation, prompt generation) — runs every Claude Code session

There's no separation because `install.sh` runs in a CI/download context without reliable `.git/` access, so everything gets deferred to session start.

## Design

### 1. Hook Declaration Convention

Vendors place hooks at a known path inside their install directory:

```
.vendored/pkg/<vendor>/hooks/
  session-start.sh    # runs on Claude Code session start (matcher: "startup")
  session-resume.sh   # runs on Claude Code session resume (matcher: "resume")
  post-install.sh     # runs after install.sh, in consumer repo context with .git/
```

All hooks are optional. A vendor declares only what it needs. Hooks must be executable shell scripts (`#!/bin/bash` or `#!/usr/bin/env bash`).

**Environment variables available to all hooks:**

| Var | Value |
|-----|-------|
| `VENDOR_NAME` | The vendor's name (e.g., `pearls`) |
| `VENDOR_PKG_DIR` | Absolute path to `.vendored/pkg/<vendor>` |
| `PROJECT_DIR` | Absolute path to the consumer repo root |

Session hooks additionally receive `CLAUDE_PROJECT_DIR` (set by Claude Code).

### 2. Hook Aggregation — Single Orchestrator

The `install` command generates a single orchestrator script at `.claude/hooks/vendored-session.sh`. This script:

- Accepts a mode argument: `--start` or `--resume`
- Calls each vendor's matching hook (`session-start.sh` or `session-resume.sh`) in **dependency order** derived from the existing `deps.json` / `.vendored/manifests/<vendor>.deps` system
- Runs post-install as a safety net (for CI-installed vendors) before session hooks, gated by a version-stamp marker
- Exits non-zero if any vendor hook fails (fail-fast)

**Generated at install time**, not discovered at runtime — the orchestrator embeds the ordered vendor list so it doesn't need to parse dependency graphs in bash.

Example generated script structure:

```bash
#!/bin/bash
set -euo pipefail
PROJECT_DIR="$(cd "$(dirname "$0")/../.." && pwd)"
MODE="${1:---start}"

# Dependency-ordered vendor list (generated by .vendored/install)
VENDORS=("git-semver" "pearls")

for vendor in "${VENDORS[@]}"; do
  PKG_DIR="$PROJECT_DIR/.vendored/pkg/$vendor"
  export VENDOR_NAME="$vendor"
  export VENDOR_PKG_DIR="$PKG_DIR"
  export PROJECT_DIR

  # Post-install safety net: run if not yet run for current version
  _run_post_install_if_needed "$vendor" "$PKG_DIR"

  # Session hook
  if [ "$MODE" = "--start" ] && [ -f "$PKG_DIR/hooks/session-start.sh" ]; then
    bash "$PKG_DIR/hooks/session-start.sh"
  elif [ "$MODE" = "--resume" ] && [ -f "$PKG_DIR/hooks/session-resume.sh" ]; then
    bash "$PKG_DIR/hooks/session-resume.sh"
  fi
done
```

### 3. `.claude/settings.json` Management — Merge Strategy

git-vendored **merges** its orchestrator entries into `.claude/settings.json` rather than owning the entire file. This lets consumers define their own non-vendor hooks.

**Merge convention:** git-vendored manages entries whose `command` contains `.claude/hooks/vendored-`. All other entries are preserved.

On regeneration:
1. Read existing `.claude/settings.json` (or `{}` if absent)
2. Filter out any existing vendored-managed entries from `hooks.SessionStart`
3. Prepend the orchestrator entries (startup + resume matchers)
4. Write back, preserving all other keys (`permissions`, etc.) and non-vendor hook entries

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "startup",
        "hooks": [{
          "type": "command",
          "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/vendored-session.sh --start",
          "statusMessage": "Configuring vendored tools..."
        }]
      },
      {
        "matcher": "resume",
        "hooks": [{
          "type": "command",
          "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/vendored-session.sh --resume",
          "statusMessage": "Configuring vendored tools..."
        }]
      }
    ]
  }
}
```

### 4. post-install.sh — Dual Execution

Runs in **two contexts** for reliability:

1. **During `install`** — The `install` command runs `post-install.sh` immediately after `install.sh` if `.git/` exists in the consumer repo. This handles local installs and CI environments with full checkouts.

2. **Session-start safety net** — The orchestrator checks a version stamp before session hooks. If the stamp doesn't match the installed version, post-install runs again. This catches CI-installed vendors where `.git/` wasn't available at install time.

**Version stamp:** `.vendored/manifests/<vendor>.post-installed` containing the version string. Compared against `.vendored/manifests/<vendor>.version`. If they differ (or stamp is missing), post-install runs and updates the stamp.

**post-install.sh contract:**
- Runs in the consumer repo root (cwd = repo root)
- Has `.git/` access
- Must be idempotent (may run multiple times)
- Receives `VENDOR_NAME`, `VENDOR_PKG_DIR`, `PROJECT_DIR` env vars

### 5. Migration from Current State

**Pearls migration:** The current `configure.sh` splits into three hooks:

| Current (configure.sh) | New location | Hook type |
|------------------------|-------------|-----------|
| Merge driver registration (`git config merge.prl-jsonl.*`) | `post-install.sh` | post-install |
| Pre-push hook symlink | `post-install.sh` | post-install |
| Create `prl` shim in `~/.local/bin` | `session-start.sh` | session-start |
| `prl prompt` (full startup) | `session-start.sh` | session-start |
| `prl prompt --resume` | `session-resume.sh` | session-resume |

After migration:
- `.claude/hooks/configure.sh` is removed from pearls' manifest
- `.claude/settings.json` is removed from pearls' manifest
- Both are managed by git-vendored framework instead
- Pearls declares hooks at `.vendored/pkg/pearls/hooks/{post-install.sh,session-start.sh,session-resume.sh}`

**Framework changes (git-vendored `install` command):**
- After running `install.sh`, scan for `post-install.sh` in vendor's hooks dir and run it
- After all vendors installed, regenerate the orchestrator and update settings.json
- The `remove` command also regenerates (to remove the vendor from the orchestrator)

## Dependency Order

Hook execution order uses the existing dependency system:

1. Read `.vendored/manifests/<vendor>.deps` for each vendor with hooks
2. Topological sort — vendors with no deps first, dependents after
3. Vendors with no dependency relationship: alphabetical tiebreak
4. Cycles: error at install time (same as existing dep resolution)

Example: if pearls depends on git-semver, git-semver's hooks run first.

## What Vendors Must Change

For a vendor to adopt session hooks:

1. Place hook scripts at `.vendored/pkg/<vendor>/hooks/` (via `$VENDOR_INSTALL_DIR/hooks/` in their `install.sh`)
2. Add hook files to their manifest output
3. Remove any direct `.claude/` file management from `install.sh`
4. That's it — the framework handles orchestration and settings.json

## Decisions

- **Hook output:** All vendor hook stdout is session context (injected into Claude's prompt). Vendors control what they emit. This preserves the current `prl prompt` behavior.
- **`.claude/settings.json`:** Merge strategy — git-vendored manages entries whose command matches `.claude/hooks/vendored-*`. Consumer-defined hooks are preserved.
- **Orchestrator:** Single generated shell script, not per-vendor settings.json entries.
- **post-install.sh:** Dual execution — during `install` (if `.git/` exists) + session-start safety net (version-stamp gated).
- **Hook order:** Dependency order from existing `deps.json`/`.deps` system, alphabetical tiebreak.

## Task Breakdown

### Phase 1: Framework — Orchestrator Generation

**gv-3adc.1 — Hook discovery and topological sort in `install`**
- Add functions to `.vendored/install`: scan `.vendored/pkg/*/hooks/` for `session-start.sh`, `session-resume.sh`, `post-install.sh`
- Topological sort using `.vendored/manifests/<vendor>.deps`
- Return ordered list of vendors with hooks

**gv-3adc.2 — Generate orchestrator script**
- `install` generates `.claude/hooks/vendored-session.sh` after all vendors are installed
- Embeds dependency-ordered vendor list
- Handles `--start` and `--resume` modes
- Includes post-install safety net with version-stamp check
- `remove` also triggers regeneration

### Phase 2: Framework — settings.json Management

**gv-3adc.3 — Merge-write `.claude/settings.json`**
- Add function to `.vendored/install`: read existing settings.json, filter vendored entries, prepend orchestrator entries, write back
- Called after orchestrator generation
- Preserves non-vendor hooks and non-hooks keys
- Creates `.claude/` directory if needed

### Phase 3: Framework — post-install.sh Execution

**gv-3adc.4 — Run post-install.sh during `install`**
- After `install.sh` completes, check for `$VENDOR_INSTALL_DIR/hooks/post-install.sh`
- Run it if `.git/` exists in consumer repo
- Write version stamp to `.vendored/manifests/<vendor>.post-installed`
- Set `VENDOR_NAME`, `VENDOR_PKG_DIR`, `PROJECT_DIR` env vars

### Phase 4: Pearls Migration

**gv-3adc.5 — Split pearls `configure.sh` into hook convention** *(pearls repo)*
- Create `hooks/post-install.sh`: merge driver registration, pre-push hook symlink
- Create `hooks/session-start.sh`: prl shim creation, `prl prompt`
- Create `hooks/session-resume.sh`: prl shim creation, `prl prompt --resume`
- Update `install.sh` to install hooks to `$VENDOR_INSTALL_DIR/hooks/` and add to manifest
- Remove `.claude/hooks/configure.sh` and `.claude/settings.json` from install.sh/manifest

### Phase 5: Cleanup

**gv-3adc.6 — Remove legacy pearls `.claude/` management from consumer repos**
- The next pearls vendor update (via `install all`) will:
  - Remove `.claude/hooks/configure.sh` (no longer in manifest)
  - Regenerate `.claude/settings.json` (framework-managed now)
  - Generate `.claude/hooks/vendored-session.sh`
- Verify the transition works end-to-end

## Open Questions (Deferred)
