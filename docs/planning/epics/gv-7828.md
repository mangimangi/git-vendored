## Context

git-vendored is the framework that manages vendor installations in consumer repos. It already writes manifest files (`.files`, `.version`, `.deps`) during install. This work extends the framework to also handle config schema files and a new audit command.

All other repos in this epic are blocked on git-vendored shipping first.

## Scope

### 1. Schema handling during install

When a vendor repo ships a `templates/config.schema` file, the framework should install it to `.vendored/manifests/<vendor>.schema` alongside existing manifest files.

The framework handles placement — vendors don't need a new env var or explicit install logic. Same pattern as `.version` and `.files`: the framework reads from the vendor source and writes to the right location.

**Acceptance criteria:**
- During `install`, if `templates/config.schema` exists in the vendor repo, copy it to `.vendored/manifests/<vendor>.schema`
- If no schema exists, skip silently (backwards compatible)
- Schema is overwritten on update (framework-managed, not user-editable)

### 2. Audit command

New command: `.vendored/audit` — validates consumer configs against installed schemas.

Separate from `check`. `check` = file protection (binary pass/fail, every PR). `audit` = config validation (advisory warnings, on-demand).

**Checks performed:**
1. **Required fields** — every required field in the schema is present in the config
2. **Unknown fields** — config has fields not in any installed vendor's schema
3. **Cross-vendor misplacement** — field X exists in vendor A's config but matches vendor B's schema
4. **Type checking** — basic type validation (string, array, object, number, boolean)

**Output format:**
```
$ python3 .vendored/audit

pearls.json
  [PASS] prefix: required field present
  [PASS] epics: valid optional field

madreperla.json
  [PASS] description: required field present
  [WARN] provider: unknown field (not in madreperla schema)

Summary: 1 warning, 0 errors
```

**Acceptance criteria:**
- `.vendored/audit` reads all `.vendored/manifests/*.schema` files
- Cross-references against all `.vendored/configs/*.json` files
- Reports required-missing as errors, unknown/misplaced as warnings
- Exit code 0 on warnings-only, non-zero on errors (missing required fields)
- Ignores `_vendor` key in configs (framework metadata, not a user field) — or `.registry` after migration

### 3. Registry migration (_vendor → manifests/.registry)

Move framework metadata from inside config files into dedicated registry files. This is Phase 2 of the migration strategy — can be implemented alongside or after schemas.

Today `_vendor` lives inside each config:
```json
{
  "_vendor": { "repo": "...", "install_branch": "...", "protected": [...] },
  "prefix": "prl"
}
```

After migration, `_vendor` content moves to `.vendored/manifests/<vendor>.registry`:
```json
{
  "repo": "mangimangi/pearls",
  "install_branch": "chore/install-pearls",
  "protected": [".pearls/**"],
  "allowed": [".pearls/issues.jsonl"]
}
```

And the config becomes purely user-owned:
```json
{
  "prefix": "prl"
}
```

**Acceptance criteria:**
- `install` writes `.vendored/manifests/<vendor>.registry` from `_vendor` data
- `check` reads from `.registry` files, falls back to `_vendor` in config for backwards compatibility
- `install` strips `_vendor` from configs on update
- Existing consumer repos with `_vendor` in configs continue to work (fallback)

### Schema format

Minimal JSON — field names, required/optional, basic types:

```json
{
  "vendor": "<vendor-name>",
  "fields": {
    "field_name": {
      "required": true,
      "type": "string",
      "description": "Human-readable description"
    }
  }
}
```

Top-level fields only. No nested field validation. Types: `string`, `number`, `boolean`, `array`, `object`.

## Cross-repo context

- Consumer repos (pearls, madreperla, git-semver, git-dogfood) will ship `templates/config.schema` — git-vendored consumes these during install
- The schema format defined here becomes the contract all vendors implement
- Consumer repos are blocked on this work shipping first

## Not in scope

- Deep JSON Schema validation
- Source layout alignment (separate follow-up epic)
- Changes to how individual tools read their own configs
